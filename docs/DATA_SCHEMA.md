# Data Schema

This document defines the JSON dataset structures and the database fields needed for experiment logging.

## Knowledge Base Files

Knowledge base files are stored in:

```txt
data/kb/kb_30.json
data/kb/kb_100.json
data/kb/kb_300.json
```

Each KB file should contain exactly the number of atomic entries indicated by its filename. An atomic entry means one independent informational unit, such as one FAQ pair, one service record, one policy rule, or one dentist profile.

Each KB file should use this top-level structure:

```json
{
  "faqs": [],
  "offerings": [],
  "policies": [],
  "dentistProfiles": []
}
```

The thesis mentions broader KB content categories. They should map into the JSON schema this way:

- FAQs: common clinic questions and answers.
- Offerings: services, products, pricing, procedure descriptions, pre-procedure preparation, and post-procedure care.
- Policies: clinic hours, appointment rules, scheduling constraints, rescheduling, cancellations, payment, insurance, and facility policies.
- Dentist profiles: dentist names, aliases, specialization, qualifications, and public profile information.

Do not duplicate or paraphrase entries just to increase KB size. The 100-entry and 300-entry files should expand the KB with distinct informational units.

## FAQ Item

```json
{
  "id": "faq_001",
  "question": "Do you accept walk-in patients?",
  "answer": "Yes, we accept walk-in patients. However, patients with prior appointments are prioritized."
}
```

## Offering Item

```json
{
  "id": "offering_001",
  "offeringType": "service",
  "name": "Oral Prophylaxis",
  "price": "PHP 800 - PHP 1,200",
  "description": "Professional dental cleaning to remove plaque, tartar, and surface stains.",
  "duration": "45 - 60 minutes"
}
```

`offeringType` must be either:

- `service`
- `product`

`duration` should be a string for services and may be `null` for products that do not have an appointment or procedure duration.

## Policy Item

```json
{
  "id": "policy_001",
  "category": "rescheduling",
  "title": "Rescheduling Notice",
  "description": "Patients are encouraged to request rescheduling at least 24 hours before their appointment."
}
```

## Dentist Profile Item

```json
{

  "aliases": ["Dra. Maria", "Doc Maria"],
  "specialization": "Orthodontics",
  "bio": "Dr. Maria Santos handles braces consultations, orthodontic adjustments, and retainer-related concerns."
}
```

## Scenario File

Scenarios are stored in:

```txt
data/scenarios/scenarios.json
```

The official experiment uses 60 fixed scenarios across all experiment conditions.

Canonical scenario format:

```json
[
  {
    "id": "scenario_001",
    "category": "general_inquiry",
    "inputType": "clean",
    "isMultiTurn": false,
    "turns": [
      {
        "turn": 1,
        "userMessage": "What time does the clinic open?",
        "expectedBehavior": "The assistant should provide the clinic operating hours based on the KB."
      }
    ]
  },
  {
    "id": "scenario_002",
    "category": "appointment_booking",
    "inputType": "noisy_taglish",
    "isMultiTurn": true,
    "turns": [
      {
        "turn": 1,
        "userMessage": "Pa appointment po sana.",
        "expectedBehavior": "The assistant should ask for the service, preferred date, or preferred time."
      },
      {
        "turn": 2,
        "userMessage": "Cleaning tom mga 3pm if meron.",
        "expectedBehavior": "The assistant should understand that the user wants a dental cleaning appointment tomorrow around 3 PM and ask for missing details or say availability must be confirmed."
      }
    ]
  }
]
```

Allowed scenario categories:

- `general_inquiry`
- `service_question`
- `appointment_booking`
- `rescheduling_cancellation`
- `ambiguous_query`

Allowed input types:

- `clean`
- `noisy_taglish`

## Multi-Turn Rules

`isMultiTurn: false` means the scenario has one user turn.

`isMultiTurn: true` means the scenario has multiple user turns.

Do not include assistant replies in the scenario file. Assistant replies must be generated by the app during the experiment.

## Database Tables

### ExperimentRun

Represents one batch experiment run.

Fields:

```txt
id
name
runType              // pilot or official
status               // pending, running, completed, failed
model
embeddingModel
ragTopK
temperature
maxOutputTokens
scenarioFileVersion
notes
startedAt
completedAt
createdAt
updatedAt
```

### ExperimentResult

Represents one generated assistant response.

Fields:

```txt
id
experimentRunId
scenarioId
scenarioCategory
inputType
isMultiTurn
turnNumber
userMessage
expectedBehavior
assistantResponse
technique            // prompt_engineering or rag
kbSize               // 30, 100, 300
latencyMs
inputTokens
outputTokens
totalTokens
retrievedContextJson // null for prompt_engineering
error
createdAt
```

Token fields use OpenAI Responses API terminology. `inputTokens`, `outputTokens`, and `totalTokens` should map to the response `usage` values.
